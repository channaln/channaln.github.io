mod_proxy_balancer vs mod_cluster: Which One Should You Use?

Oct 26, 2023 · Channa Sandaruwan · Apache load balancing

When setting up load balancing in front of a JBoss EAP or Tomcat cluster, Apache HTTP Server gives you two built-in paths: mod_proxy_balancer (the classic approach) and mod_cluster (the JBoss-native, dynamic approach). Both distribute traffic across backend nodes, but their architectures are fundamentally different and picking the wrong one will hurt you at scale or during maintenance windows.

How mod_proxy_balancer Works

With mod_proxy_balancer, the entire configuration lives in Apache. You list your backend workers statically, choose a balancing algorithm, and Apache does the rest. The backend servers have zero knowledge that a load balancer exists in front of them.

Three balancing algorithms are available:

  • byrequests (default) distributes request counts equally over time.
  • bytraffic distributes by bytes transferred; useful when response sizes vary dramatically.
  • bybusyness routes to whichever worker currently has the fewest active requests. The best choice for most application servers.
<Proxy "balancer://mycluster">
 BalancerMember http://node1:8080 route=node1
 BalancerMember http://node2:8080 route=node2
 BalancerMember http://node3:8080 route=node3
 ProxySet lbmethod=bybusyness
</Proxy>

ProxyPass "/" "balancer://mycluster/"
ProxyPassReverse "/" "balancer://mycluster/"

# Sticky sessions via JSESSIONID route
Header add Set-Cookie "ROUTEID=.%{BALANCER_WORKER_ROUTE}e; path=/" env=BALANCER_ROUTE_CHANGED

How mod_cluster Works

mod_cluster inverts the model entirely. Rather than Apache knowing about the backends, each JBoss EAP or Tomcat node advertises itself to Apache on startup using MCMP (Mod-Cluster Management Protocol). When a new node comes up, it registers and when it shuts down gracefully, it de-registers, draining existing sessions before leaving the pool. Apache's config never needs to change.

The load metrics that influence routing decisions come directly from the JVM: active session count, system CPU load, and heap usage. This means Apache makes routing decisions based on what the backend actually knows about its own health, not just a request counter.

<!-- JBoss EAP standalone.xml mod_cluster subsystem -->
<subsystem xmlns="urn:jboss:domain:modcluster:5.0">
 <proxy name="default"
 advertise="true"
 advertise-socket="modcluster"
 proxies="main-proxy">
 <dynamic-load-provider>
 <load-metric type="busyness" weight="2"/>
 <load-metric type="heap" weight="1"/>
 </dynamic-load-provider>
 </proxy>
</subsystem>

On the Apache side, you only need to enable MCMP reception you don't list any backend workers:

LoadModule advertise_module modules/mod_advertise.so
LoadModule manager_module modules/mod_manager.so
LoadModule proxy_cluster_module modules/mod_proxy_cluster.so

<VirtualHost *:6666>
 <Directory />
 Require ip 10.0.0.0/8
 </Directory>
 ManagerBalancerName mycluster
 AdvertiseFrequency 5
 EnableMCPMReceive
</VirtualHost>

ProxyPass / balancer://mycluster/
ProxyPassReverse / balancer://mycluster/

When to Use Each

Use mod_proxy_balancer when:

  • Your backends are heterogeneous Node.js, Python, Nginx, mixed technologies.
  • You want the simplest possible setup with minimal moving parts.
  • Your cluster size is fixed and rarely changes.
  • You're proxying to non-Java applications that don't support MCMP.

Use mod_cluster when:

  • Your backends are JBoss EAP or Tomcat this is what mod_cluster was designed for.
  • You scale nodes in and out dynamically (autoscaling, rolling deployments).
  • Graceful rolling restarts without dropped sessions are critical to your deployment pipeline.
  • You want routing decisions to reflect actual JVM health, not just request counts.
In production at KUWAITNET, we use mod_cluster for all JBoss EAP environments. The automatic drain behaviour alone eliminated session loss during rolling upgrades something we previously had to orchestrate manually with cron jobs and shell scripts.

Conclusion

Both modules are solid and well-maintained. The decision really comes down to your backend: if it's a JBoss EAP or Tomcat cluster, mod_cluster's dynamic registration and JVM-aware load metrics are worth the slightly higher setup cost. If you're balancing a mixed or non-Java backend, mod_proxy_balancer's simplicity wins every time.
