JBoss Domain Mode A Comprehensive Guide

Oct 26, 2023 · Channa Sandaruwan · JBoss EAP

Most JBoss EAP deployments start in standalone mode one process, one config file, easy to reason about. Domain mode introduces a fundamentally different operational model. It exists specifically to solve one problem: how do you manage many JBoss instances as a coherent unit without logging into each one individually? When you have dozens of nodes, domain mode pays for its complexity many times over.

Standalone vs Domain Mode

In standalone mode, each server is an island. It has its own standalone.xml, and to push a configuration change to ten servers you edit ten files. Domain mode introduces a single control point: the Domain Controller. All configuration lives centrally in domain.xml, and changes propagate automatically to all managed servers.

AspectStandalone ModeDomain Mode
Config filesstandalone.xml per serverdomain.xml (shared) + host.xml per host
ManagementPer-server management portSingle domain controller entry point
DeploymentPer-server deploy commandDeploy once to a server group
Rolling restartManual orchestrationBuilt-in rolling-restart-servers command
Suitable forDevelopment, single-node productionMulti-server production clusters

Domain Mode Architecture

Three tiers make up the domain mode architecture:

Domain Controller (DC)

The master process. It hosts domain.xml and its own host.xml (the master host configuration, sometimes called host-master.xml). All management operations deployments, configuration changes, server restarts flow through the DC. There is exactly one domain controller per domain.

Host Controller (HC)

Runs on every physical or virtual machine that participates in the domain. Connects to the DC on startup, receives its configuration from domain.xml, and manages the JBoss server instances on its local host. Each host has its own host.xml (host-slave.xml), which defines which servers run on that machine and how to reach the domain controller.

Server Instance

The actual JBoss EAP process. Each server belongs to a Server Group and receives its subsystem configuration (logging, datasources, security, messaging) from the profile assigned to that server group.

Server Group

A logical grouping of server instances that share a configuration profile and socket binding group. You deploy applications to a server group, and every server in that group receives the deployment automatically.

Key Configuration Files

domain.xml Owned by the Domain Controller

<domain>
 <!-- Profiles: named sets of subsystem configurations -->
 <profiles>
 <profile name="full">
 <subsystem xmlns="urn:jboss:domain:logging:8.0"> ... </subsystem>
 <subsystem xmlns="urn:jboss:domain:datasources:6.0"> ... </subsystem>
 <!-- ... more subsystems ... -->
 </profile>
 </profiles>

 <!-- Socket binding groups: port definitions -->
 <socket-binding-groups>
 <socket-binding-group name="full-sockets"
 default-interface="public">
 <socket-binding name="http" port="${jboss.http.port:8080}"/>
 <socket-binding name="https" port="${jboss.https.port:8443}"/>
 </socket-binding-group>
 </socket-binding-groups>

 <!-- Server groups: bind profile + socket-group + JVM settings -->
 <server-groups>
 <server-group name="main-server-group" profile="full">
 <jvm name="default">
 <heap size="512m" max-size="1024m"/>
 <jvm-options>
 <option value="-XX:+UseG1GC"/>
 </jvm-options>
 </jvm>
 <socket-binding-group ref="full-sockets"/>
 </server-group>
 </server-groups>
</domain>

host.xml Owned by Each Host Controller

<host name="host1" xmlns="urn:jboss:domain:20.0">

 <!-- How to reach the domain controller -->
 <domain-controller>
 <remote host="${jboss.domain.master.address:DC_HOSTNAME}"
 port="${jboss.domain.master.port:9990}"
 authentication-context="hc-auth-context"/>
 </domain-controller>

 <!-- Servers running on this host -->
 <servers>
 <server name="server-one" group="main-server-group" auto-start="true">
 <socket-bindings port-offset="0"/>
 </server>
 <server name="server-two" group="main-server-group" auto-start="true">
 <socket-bindings port-offset="150"/>
 <!-- server-two uses ports 8080+150=8230, 8443+150=8593, etc. -->
 </server>
 </servers>

</host>

The port offset is critical when running multiple servers on the same host it shifts every socket binding by that amount, preventing port conflicts without requiring a complete redefinition of the socket binding group.

Starting a Domain

# Start the domain controller (on the DC host)
./domain.sh --host-config=host-master.xml

# Start a host controller (on each slave host)
./domain.sh \
 --host-config=host-slave.xml \
 -Djboss.domain.master.address=DC_HOSTNAME

Deploying Applications

With the domain running, deploy to a server group all servers in the group receive the deployment automatically:

# Connect to domain controller CLI
./jboss-cli.sh --connect --controller=DC_HOST:9990

# Deploy to a server group
deploy /path/to/myapp.war --server-groups=main-server-group

# Verify deployment status on all servers in the group
/server-group=main-server-group/deployment=myapp.war:read-resource(include-runtime=true)

# Undeploy from a server group
undeploy myapp.war --server-groups=main-server-group

Rolling Restart

One of domain mode's most valuable operational capabilities is the coordinated rolling restart. JBoss EAP restarts servers in the group sequentially, ensuring the application remains available throughout:

# Rolling restart restarts each server one at a time
/server-group=main-server-group:rolling-restart-servers(blocking=true)

# Reload all servers (faster only applies config changes without full restart)
/server-group=main-server-group:reload-servers(blocking=false)

# Restart a single server on a specific host
/host=host1/server=server-one:restart()

Useful CLI Tips

# Check status of all servers across the domain
/host=*/server=*:read-resource(include-runtime=true)

# Read all running server names on host1
/host=host1:read-children-names(child-type=server)

# Force-stop a server that isn't responding
/host=host1/server=server-two:stop(timeout=30)

# Add a system property to a server group (all servers inherit it)
/server-group=main-server-group/system-property=my.prop:add(value="production")

When Domain Mode Is Worth It

Domain mode earns its complexity when:

  • You have 3 or more JBoss EAP instances that need to stay in sync.
  • You need zero-downtime rolling deployments coordinated automatically.
  • You want to enforce a single source of truth for subsystem configuration datasources, messaging, security, logging across all nodes.
  • You're running JBoss alongside a load balancer (HAProxy, F5, or Apache with mod_cluster) and need graceful rolling upgrades without manual load balancer coordination.
Domain mode and mod_cluster together create a self-managing cluster: nodes register with the load balancer automatically on startup, drain and deregister on shutdown, and rolling restarts are a single CLI command. That combination is what makes JBoss EAP production deployments genuinely manageable at scale.
