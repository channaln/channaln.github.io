Upgrading Apache Tomcat — A Practical Guide

Oct 26, 2023 · Channa Sandaruwan · Middleware

Tomcat upgrades look deceptively simple — extract a new archive, copy your config, done. In practice, each major version introduces deprecated attributes, changed defaults, and occasionally breaking changes that will bring down your applications if you're not prepared. This guide documents the process I use in production, built from managing Tomcat environments across multiple enterprise deployments.

Why You Should Upgrade

Running an outdated Tomcat version means missing security patches for actively exploited CVEs. Beyond security, newer releases bring meaningful performance improvements in HTTP/2 handling, connection threading, and memory management. The longer you delay, the harder the eventual jump becomes — especially crossing the Tomcat 9 → 10 boundary where the Jakarta EE namespace migration is required.

Pre-Upgrade Checklist

Before downloading anything, confirm:

  • Servlet spec compatibility: Tomcat 9 = Servlet 4.0 (javax.*). Tomcat 10+ = Jakarta EE 9+ (jakarta.*). Crossing this boundary requires application-level changes.
  • Java version: Each Tomcat version has a minimum required JDK. Tomcat 10.1 requires Java 11 or later.
  • Review the changelog for deprecated attributes or removed features that affect your server.xml configuration.
  • Inventory all custom config: server.xml, context.xml, web.xml, setenv.sh, and any custom Valves or Realms.

Step 1 — Full Backup

Never skip this step. Back up the entire installation directory and your deployment artifacts:

# Full installation backup
cp -a /opt/tomcat /opt/tomcat-bkp-$(date +%Y%m%d)

# Config-only backup (faster to restore)
tar czf /backup/tomcat-conf-$(date +%Y%m%d).tar.gz \
  /opt/tomcat/conf/ \
  /opt/tomcat/bin/setenv.sh \
  /opt/tomcat/webapps/

# Record current version
/opt/tomcat/bin/version.sh | head -3

Step 2 — Download and Verify

Always download from the official Apache mirror and verify the checksum before extracting:

# Download (adjust version number)
wget https://downloads.apache.org/tomcat/tomcat-10/v10.1.28/bin/apache-tomcat-10.1.28.tar.gz
wget https://downloads.apache.org/tomcat/tomcat-10/v10.1.28/bin/apache-tomcat-10.1.28.tar.gz.sha512

# Verify integrity
sha512sum -c apache-tomcat-10.1.28.tar.gz.sha512

Step 3 — Side-by-Side Installation

Install the new version alongside the existing one. Never overwrite an active installation:

tar xzf apache-tomcat-10.1.28.tar.gz -C /opt/
# At this point you have both:
#   /opt/tomcat             (current symlink → old version)
#   /opt/apache-tomcat-10.1.28  (new version, not yet active)

Step 4 — Migrate Configuration

This is the highest-risk step. Do not copy configuration files wholesale — always diff and merge:

# Diff each file individually
diff /opt/tomcat/conf/server.xml /opt/apache-tomcat-10.1.28/conf/server.xml
diff /opt/tomcat/conf/context.xml /opt/apache-tomcat-10.1.28/conf/context.xml
diff /opt/tomcat/conf/web.xml /opt/apache-tomcat-10.1.28/conf/web.xml

Key things to watch per file:

  • server.xml — Check HTTP connector attributes. In Tomcat 9+, scheme and secure on the HTTP connector were deprecated in favour of RemoteIpValve. SSL connector attribute names changed significantly in Tomcat 9.
  • context.xml — Resource definitions and session persistence configuration. These are usually safe to copy directly.
  • web.xml — The default servlet and JSP servlet mappings changed between versions. Merge your custom entries into the new file rather than replacing it.
  • setenv.sh — Safe to copy; just review JVM flags for compatibility with the new JDK requirement.

Copy your merged configuration to the new installation:

cp /opt/tomcat/conf/server.xml     /opt/apache-tomcat-10.1.28/conf/
cp /opt/tomcat/conf/context.xml    /opt/apache-tomcat-10.1.28/conf/
cp /opt/tomcat/bin/setenv.sh       /opt/apache-tomcat-10.1.28/bin/
cp -r /opt/tomcat/webapps/myapp    /opt/apache-tomcat-10.1.28/webapps/

Step 5 — Stop, Switch, Start

# Graceful shutdown of current instance
/opt/tomcat/bin/shutdown.sh
sleep 10  # Wait for in-flight requests to complete

# Switch the symlink atomically
ln -sfn /opt/apache-tomcat-10.1.28 /opt/tomcat

# Start the new instance
/opt/tomcat/bin/startup.sh

# Watch startup logs
tail -f /opt/tomcat/logs/catalina.out

Step 6 — Validate

After startup, run through these checks before declaring success:

  • All applications deployed without errors — check catalina.out for SEVERE or ERROR log lines.
  • Test every application endpoint, including authenticated paths.
  • Verify SSL/TLS connectors: openssl s_client -connect hostname:8443 and confirm the certificate chain.
  • Check thread pool health via the Manager app at /manager/status.
  • Run your application's smoke test suite if you have one.

Rollback Procedure

If anything is wrong, rollback is a two-line operation because of the symlink approach:

# Stop the new instance
/opt/tomcat/bin/shutdown.sh

# Restore the symlink to the old version
ln -sfn /opt/apache-tomcat-<old-version> /opt/tomcat

# Start the old instance
/opt/tomcat/bin/startup.sh

Common Issues I've Hit in Production

IssueCauseFix
ClassNotFoundException: javax.servlet.*Application uses javax.* but Tomcat 10+ expects jakarta.*Use Tomcat Migration Tool or stay on Tomcat 9.x
SSL connector rejects startDeprecated attributes in server.xmlReplace old-style SSL config with newer NIO connector + SSLHostConfig
Applications fail to deploy silentlyweb.xml version mismatchCheck <web-app version="..."> matches your servlet spec
OutOfMemoryError on startupmaxThreads or heap settings not migrated from setenv.shVerify setenv.sh was copied and is executable
The most expensive mistake you can make is upgrading Tomcat and your application at the same time. Change one variable at a time — upgrade the container first, validate, then update the application.
