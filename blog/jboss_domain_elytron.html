Elytron in JBoss Domain Mode Security Realm Configuration

Oct 26, 2023 · Channa Sandaruwan · JBoss EAP

Configuring Elytron in standalone mode is reasonably well documented. Doing the same in domain mode adds complexity that catches many engineers off guard security configuration is split between domain.xml and host.xml, and the rules for which belongs where are not immediately obvious. This article explains the split, walks through adding an Elytron realm to a domain profile, and shows how to manage it via CLI.

Where Security Lives in Domain Mode

The first thing to understand is the responsibility boundary:

  • domain.xml contains profiles. The Elytron subsystem inside a profile controls security for applications running on servers in server groups that use that profile. This is where you define realms, security domains, and HTTP authentication factories.
  • host.xml contains host-specific configuration. The management security realm (which controls access to the CLI and admin console) lives here, separately from application security. Each host controller has its own management realm configuration.

This separation means you can run the same application security configuration across all nodes (since all servers in a server group share a profile in domain.xml), while each host's management access is controlled independently.

Elytron in a Domain Profile (domain.xml)

Open domain.xml and locate the profile your server group uses (commonly full or full-ha). Within that profile, find or create the Elytron subsystem:

<!-- domain.xml inside <profile name="full"> -->
<subsystem xmlns="urn:wildfly:elytron:16.0">

 <security-realms>
 <properties-realm name="ApplicationRealm">
 <users-properties
 path="application-users.properties"
 relative-to="jboss.domain.config.dir"
 digest-realm-name="ApplicationRealm"/>
 <groups-properties
 path="application-roles.properties"
 relative-to="jboss.domain.config.dir"/>
 </properties-realm>
 </security-realms>

 <security-domains>
 <security-domain name="ApplicationDomain"
 default-realm="ApplicationRealm"
 permission-mapper="default-permission-mapper">
 <realm name="ApplicationRealm" role-decoder="groups-to-roles"/>
 </security-domain>
 </security-domains>

 <http>
 <http-authentication-factory name="application-http-authentication"
 http-server-mechanism-factory="global"
 security-domain="ApplicationDomain">
 <mechanism-configuration>
 <mechanism mechanism-name="BASIC">
 <mechanism-realm realm-name="ApplicationRealm"/>
 </mechanism>
 <mechanism mechanism-name="FORM"/>
 </mechanism-configuration>
 </http-authentication-factory>
 </http>

</subsystem>

Because this lives inside a profile, every server in every server group that uses the full profile will automatically have this security configuration. You don't need to configure each server individually.

Adding a New Realm via CLI

Connect to the domain controller's management port and use the profile path prefix for all commands:

# Connect to domain controller
./jboss-cli.sh --connect --controller=DC_HOST:9990

# Add a new properties realm to the 'full' profile
/profile=full/subsystem=elytron/properties-realm=MyRealm:add(\
 users-properties={path=my-users.properties, \
 relative-to=jboss.domain.config.dir, \
 digest-realm-name=MyRealm}, \
 groups-properties={path=my-roles.properties, \
 relative-to=jboss.domain.config.dir})

# Create a security domain using the new realm
/profile=full/subsystem=elytron/security-domain=MyDomain:add(\
 default-realm=MyRealm, \
 permission-mapper=default-permission-mapper, \
 realms=[{realm=MyRealm, role-decoder=groups-to-roles}])

# Create an HTTP authentication factory
/profile=full/subsystem=elytron/http-authentication-factory=my-http-auth:add(\
 http-server-mechanism-factory=global, \
 security-domain=MyDomain, \
 mechanism-configurations=[{\
 mechanism-name=FORM},{\
 mechanism-name=BASIC, \
 mechanism-realm-configurations=[{realm-name=MyRealm}]}])

Management Interface Security (host.xml)

The management interface security is configured in host.xml per host controller. By default it uses a legacy security realm, but you can migrate it to Elytron:

<!-- host.xml management interfaces -->
<management-interfaces>
 <http-interface http-authentication-factory="management-http-authentication">
 <http-upgrade enabled="true"
 sasl-authentication-factory="management-sasl-authentication"/>
 <socket interface="management" port="${jboss.management.http.port:9990}"/>
 </http-interface>
</management-interfaces>

The management authentication factories (management-http-authentication and management-sasl-authentication) are also defined in the Elytron subsystem in host.xml, separate from the profile-level configuration in domain.xml.

Propagating Changes to Running Servers

After modifying a profile's Elytron configuration, servers that use that profile will need to reload to pick up the changes. In domain mode, you can reload all servers in a server group with a single command:

# Check which servers are in reload-required state
/server-group=main-server-group:read-children-resources(child-type=deployment)

# Reload all servers in the group (non-blocking servers restart one by one)
/server-group=main-server-group:reload-servers(blocking=false)

# Or reload a specific server on a specific host
/host=host1/server=server-one:reload()

Common Pitfalls

  • Wrong config file: If your realm change has no effect, double-check which file it's in. Application security goes in domain.xml (profile), not host.xml.
  • Properties files must exist on each host: If you use properties-realm with relative-to=jboss.domain.config.dir, the properties files must be present on every host controller's config directory they are not distributed automatically by the domain controller.
  • Profile changes don't reload automatically: Always explicitly reload the server group after Elytron configuration changes, otherwise servers continue using the old configuration from their last startup.
For production domain setups with LDAP integration, define a separate profile for production (e.g., full-prod) that uses your LDAP realm, and keep a full-dev profile with properties-based realms for development and testing. Server groups can reference different profiles with zero code changes.
