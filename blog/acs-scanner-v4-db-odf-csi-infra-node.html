When OpenShift Storage Placement Breaks ACS Scanner V4 DB

Apr 30, 2026 · Channa Sandaruwan · OpenShift Storage

Some OpenShift issues look like an application problem at first. The pod is stuck, the operator is waiting, and the error message points at a PVC. But the real problem is hidden one layer lower: the node where the pod landed is not ready to consume that storage.

I recently hit this in an OpenShift 4.18 cluster with a dedicated node layout:

  • 6 worker nodes
  • 2 infra nodes
  • 4 ODF storage nodes
  • Infra nodes tainted for infrastructure workloads
  • Storage nodes tainted for ODF
  • Ingress, ACM, and ACS workloads required to run only on infra nodes

The installation order was straightforward: ODF first, then ACM, then ACS. ACS Central was configured with node placement and tolerations so the workloads could run on the infra nodes. That part looked correct.

Then Scanner V4 DB got stuck.

The Symptom

The Scanner V4 DB pod stayed in ContainerCreating. The pod was scheduled on an infra node, but the PVC mount failed with this error:

MountVolume.MountDevice failed for volume
"pvc-f3d4be76-e415-4600-a12f-82410bcdd277":
kubernetes.io/csi: attacher.MountDevice failed
to create newCsiDriverClient: driver name
openshift-storage.rbd.csi.ceph.com not found
in the list of registered CSI drivers

At first glance, this looks like an ACS installation problem or a PVC problem. But the important clue is this part:

driver name openshift-storage.rbd.csi.ceph.com not found

That means the pod landed on a node where the required CSI driver was not registered.

The Real Problem

The ACS workload was allowed to run on the infra node because the ACS tolerations were configured.

But the ODF RBD CSI plugin was not running on that infra node.

That is the trap.

When a pod uses an ODF-backed PVC, the node running the pod must have the required CSI plugin available. It is not enough for ODF to be healthy on the storage nodes. The consumer node also needs the CSI plugin DaemonSet running locally.

In this case, the infra node had a custom taint. The ODF CSI plugin DaemonSet did not tolerate that taint by default, so the csi-rbdplugin pod was never scheduled there.

ACS Scanner V4 DB scheduled on infra node
        +
ODF PVC required
        +
ODF RBD CSI plugin missing on infra node
        =
PVC mount failure

Why It Was Easy To Miss

The scheduling requirement was valid: ACS, ACM, and ingress had to run on infra nodes.

The ACS tolerations were also valid: they allowed the ACS pods to land where they were supposed to land.

But storage has its own placement requirement. A workload using a PVC needs the storage driver on the same node. So whenever we force a persistent workload onto tainted infra nodes, we must also ask: can the CSI plugin run there?

What To Check

First, check where the failing Scanner V4 DB pod is scheduled:

oc get pod -n stackrox -o wide | grep scanner

Then check whether the ODF RBD CSI plugin is running on that same node:

oc get pods -n openshift-storage -o wide | grep csi-rbdplugin

Check the pod events:

oc describe pod -n stackrox <scanner-v4-db-pod-name>

And confirm the node taints:

oc describe node <infra-node-name> | grep -i taint -A2

The Fix

The fix is to allow the ODF CSI plugin DaemonSet to tolerate the infra node taint.

Edit the rook-ceph operator config map:

oc edit configmap rook-ceph-operator-config -n openshift-storage

Add the toleration under CSI_PLUGIN_TOLERATIONS.

Example for an infra role taint:

data:
  CSI_PLUGIN_TOLERATIONS: |
    - key: node-role.kubernetes.io/infra
      operator: Exists
      effect: NoSchedule
    - key: node.ocs.openshift.io/storage
      operator: Equal
      value: "true"
      effect: NoSchedule

One important detail: operator: Equal must include the value you expect. For the usual ODF storage taint, this is correct:

- key: node.ocs.openshift.io/storage
  operator: Equal
  value: "true"
  effect: NoSchedule

If you only want to tolerate the key regardless of value, use Exists:

- key: node.ocs.openshift.io/storage
  operator: Exists
  effect: NoSchedule

Using operator: Equal without a value is not the right shape because there is no value to compare.

If your cluster uses a custom infra taint, match the real key and value:

data:
  CSI_PLUGIN_TOLERATIONS: |
    - key: nodetype
      operator: Equal
      value: "infra"
      effect: NoSchedule
    - key: node.ocs.openshift.io/storage
      operator: Equal
      value: "true"
      effect: NoSchedule

If the plugin pods do not appear automatically, restart the rook-ceph operator pod:

oc delete pod -n openshift-storage -l app=rook-ceph-operator

Validation

After the change, verify the RBD CSI plugin is running on the infra nodes:

oc get pods -n openshift-storage -o wide | grep csi-rbdplugin

Then check the Scanner V4 DB pod again:

oc get pods -n stackrox -o wide | grep scanner

The pod should move past ContainerCreating once the PVC can be mounted.

Why This DaemonSet Matters

The csi-rbdplugin DaemonSet is not just another storage pod. It is the node-side part of the CSI driver. It registers the RBD CSI driver on the node and performs the mount work when a pod on that node uses an ODF RBD PVC.

That is why the node role does not matter as much as the workload placement.

If a pod can run on a node and that pod can mount an ODF PVC, then that node needs the ODF CSI node plugin. It can be a normal worker node, an infra node, or any other worker-role node. The node does not need to be an ODF storage node, but it does need the CSI plugin to consume ODF storage.

This is the key difference:

  • Storage nodes run the ODF storage services.
  • Consumer nodes run application pods that use ODF PVCs.
  • CSI node plugins must run on consumer nodes so the volumes can be mounted.

So yes, in a normal cluster where workloads can run on all workers, the CSI plugin should be available on all workers. In a cluster with tainted infra nodes, the CSI plugin also needs the toleration for those taints if ACS, ACM, ingress, or any other persistent workload is forced to run there.

Lesson Learned

This was not only an ACS issue, and it was not only an ODF issue. It was a placement design issue.

When we design OpenShift clusters with dedicated infra and storage nodes, we usually think about where the application pods should run. But for persistent workloads, we also need to think about where the storage plugins must run.

If a tainted node will run pods that consume ODF PVCs, the ODF CSI plugin must tolerate that taint and run on that node.

That one check can save a lot of time when troubleshooting ContainerCreating pods with CSI mount errors.
