Elytron Security Realm — What It Is and Why It Matters

Oct 26, 2023 · Channa Sandaruwan · Security

If you've worked with JBoss EAP for a while, you've seen the transition from the old PicketBox-based security configuration to Elytron. The change is significant — not just syntactically, but architecturally. Understanding what Elytron is and how its pieces fit together is the foundation for securing JBoss EAP applications effectively in modern deployments.

The Problem Elytron Solved

PicketBox, the security framework JBoss EAP used until version 7.3, was built around JAAS (Java Authentication and Authorisation Service) with JBoss-specific extensions layered on top. This worked, but it had deep structural problems:

  • Authentication and authorisation logic were tightly coupled and hard to test independently.
  • Extending the framework required implementing JBoss-specific SPIs that weren't standardised.
  • Security configurations were scattered across multiple subsystems and XML stanzas with little coherence.
  • Auditing security decisions was difficult because the responsibility chain was implicit.

Elytron (named after the protective wing casings of beetles — a fitting metaphor for a protective security layer) was rebuilt from scratch with three explicit goals: separation of concerns, composability, and programmatic access. Everything in Elytron is a named, independently configurable component, and you wire components together explicitly.

Core Architecture

Elytron breaks security into distinct composable primitives. Each one does exactly one thing:

Security Realm

A realm answers a single question: does this identity exist, and what is their credential? It is the source of identity data. Elytron ships several realm types:

  • properties-realm — reads users and role memberships from flat .properties files. Simplest option; good for development and small deployments.
  • filesystem-realm — stores each identity as a file on disk under a directory hierarchy. Scales better than properties files.
  • ldap-realm — queries an LDAP or Active Directory server for user data and group memberships. The standard choice for enterprise environments.
  • jdbc-realm — loads identity data from a relational database via JDBC. Useful when your user store is already a database.
  • token-realm — validates OAuth2/OIDC JWT tokens. Used in API security and microservice deployments.
  • aggregate-realm — combines multiple realms. A common pattern: load credentials from LDAP but read roles from a database.

Security Domain

A security domain combines one or more realms with policy decisions. It is what applications associate with — not realms directly. The security domain defines which realm(s) to use for authentication, how to decode roles from identity attributes, and what permissions to assign.

Authentication Factory

Wraps a security domain with specific authentication mechanisms for a protocol. There are two varieties:

  • http-authentication-factory — for HTTP applications (BASIC, FORM, DIGEST, CLIENT_CERT).
  • sasl-authentication-factory — for management interfaces and EJB remoting (PLAIN, SCRAM-SHA-256, GSSAPI).

Role Decoder

Maps attributes from the identity (such as an LDAP groups attribute or a database column) to application roles. The most common decoder is groups-to-roles, which maps the groups attribute directly to role names.

Permission Mapper

Decides what permissions an authenticated principal is granted. Most configurations use default-permission-mapper, which grants LoginPermission to all successfully authenticated identities.

A Minimal Working Example

Here is a complete, minimal Elytron configuration in standalone.xml that secures an HTTP application using a properties file-backed realm:

<subsystem xmlns="urn:wildfly:elytron:16.0">

    <!-- 1. Define the realm (source of identities) -->
    <security-realms>
        <properties-realm name="AppRealm">
            <users-properties path="app-users.properties"
                              relative-to="jboss.server.config.dir"
                              digest-realm-name="AppRealm"/>
            <groups-properties path="app-roles.properties"
                               relative-to="jboss.server.config.dir"/>
        </properties-realm>
    </security-realms>

    <!-- 2. Wire realm into a security domain -->
    <security-domains>
        <security-domain name="AppDomain"
                         default-realm="AppRealm"
                         permission-mapper="default-permission-mapper">
            <realm name="AppRealm" role-decoder="groups-to-roles"/>
        </security-domain>
    </security-domains>

    <!-- 3. Create an HTTP authentication factory -->
    <http>
        <http-authentication-factory name="app-http-auth"
                                     http-server-mechanism-factory="global"
                                     security-domain="AppDomain">
            <mechanism-configuration>
                <mechanism mechanism-name="FORM"/>
                <mechanism mechanism-name="BASIC">
                    <mechanism-realm realm-name="AppRealm"/>
                </mechanism>
            </mechanism-configuration>
        </http-authentication-factory>
    </http>

</subsystem>

The corresponding app-users.properties file (passwords are stored as digests in format user=HEX(MD5(user:realm:password))):

# app-users.properties
admin=a4d0ccfe1523e901a3a0c0e60a2a7e65

# Generate new entries with:
# $JBOSS_HOME/bin/add-user.sh -a -u admin -p MyPassword -r AppRealm

Referencing Elytron from Your Application Deployment

Once the authentication factory is defined, reference it in your application deployment or in the Undertow subsystem's application-security-domain mapping:

<subsystem xmlns="urn:jboss:domain:undertow:12.0">
    <application-security-domains>
        <application-security-domain name="AppDomain"
            http-authentication-factory="app-http-auth"/>
    </application-security-domains>
</subsystem>

Your web.xml security constraints then work against role names defined in app-roles.properties without any further changes.

Why the Composable Model Matters

In a PicketBox world, changing your identity store from a properties file to LDAP required touching multiple configuration points and often application-level changes. With Elytron, you swap the properties-realm for an ldap-realm and nothing else changes — the security domain, the authentication factory, the Undertow mapping, and the application are all untouched.

Think of Elytron as a pipeline: realm authenticates the identity → role decoder maps attributes → security domain enforces policy → authentication factory exposes the mechanism. Each stage is independently replaceable.

This design also makes auditing clean. When a security incident occurs, you can trace exactly which component made each decision: which realm authenticated the user, which decoder mapped their groups, and which permission mapper granted access. In PicketBox, that trace was buried in opaque JAAS callbacks.
